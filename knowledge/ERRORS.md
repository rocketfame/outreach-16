## [2026-07-15] Automation API: три системні дефекти, знайдені зовнішнім black-box probing
**Причина:** (1) `202 {"status":"queued"}` брехав — слот перевірявся тільки в `after()`, друга джоба тихо вмирала з `concurrency_limit`; (2) `language` приймався, але ігнорувався — `pipeline.ts` хардкодив `language: "English"`, іспанський запит гарантовано давав англійську статтю з 200 OK; (3) усі automation-джоби ділили один rate-limit бакет `generate` 10/год через `x-forwarded-for: "automation"` — батч на 10-12 статей впирався в ліміт із загадковими падіннями.
**Рішення:** справжня FIFO-черга в KV з опортуністичним дренажем (кожен POST/GET-poll — тригер), атомарні слоти SET NX + one-shot started-guard; language валідується проти `config/languages.ts` і прокидається в brief; внутрішні виклики обходять rate limiter через in-process токен (`lib/automation/internal.ts`).
**Урок:** статус у відповіді — це контракт: якщо кажеш "queued", джоба мусить колись виконатись. Параметри, що приймаються без валідації, гірші за відхилені — мовчазна деградація виявляється лише коли людина читає драфт. І get-then-set на Redis — завжди race: тільки SET NX.
